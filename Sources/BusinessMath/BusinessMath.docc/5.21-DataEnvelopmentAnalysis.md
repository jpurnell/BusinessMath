# Data Envelopment Analysis

Evaluate the relative efficiency of decision-making units across multiple inputs and outputs using linear programming.

## Overview

Data Envelopment Analysis (DEA) answers a deceptively simple question: *given a set of comparable entities, which ones are using their resources most efficiently?*

Unlike weighted scoring, DEA does not require you to pre-assign importance weights to inputs or outputs. Instead, it gives each entity the benefit of the doubt — finding whatever weight combination makes it look best. If an entity is still inefficient under its own best-case weights, that's a genuine finding.

### When to Use DEA

DEA is the right tool when you need to:
- Compare entities (branches, products, hospitals, funds) across **multiple inputs and outputs** simultaneously
- Identify which entities are on the efficient frontier and which are not
- Quantify **how much** an inefficient entity must improve, and in which dimensions
- Rank among efficient entities (via super-efficiency)
- Get **dimension-specific** improvement targets (via the Slacks-Based Measure)

### Quick Example

```swift
import BusinessMath

let dmus = [
    DMU(name: "Branch A", inputs: [50, 20], outputs: [100, 80]),
    DMU(name: "Branch B", inputs: [30, 30], outputs: [90, 70]),
    DMU(name: "Branch C", inputs: [60, 10], outputs: [110, 60]),
    DMU(name: "Branch D", inputs: [55, 25], outputs: [85, 65])
]

let solver = DEASolver()
let result = try solver.solve(dmus: dmus, model: .ccr)

for score in result.scores {
    let status = score.isEfficient ? "EFFICIENT" : "score: \(score.efficiency)"
    print("\(score.name): \(status)")
}
```

## CCR Model (Constant Returns to Scale)

The foundational DEA model from Charnes, Cooper, and Rhodes (1978). Assumes that scaling inputs proportionally produces proportional outputs. Use CCR when the entities being compared operate at similar scale, or when scale differences should be penalized.

```swift
let result = try solver.solve(dmus: dmus, model: .ccr)
```

**Input-oriented** (default): Asks "by how much can this entity reduce its inputs while maintaining current outputs?" A score of 0.75 means the entity could produce the same output with 75% of its current inputs.

**Output-oriented**: Asks "by how much can this entity increase its outputs using current inputs?" Scores are internally normalized to (0, 1] — the raw LP value is available via ``DMUScore/rawScore``.

```swift
let result = try solver.solve(
    dmus: dmus,
    model: .ccr,
    orientation: .outputOriented
)
```

## BCC Model (Variable Returns to Scale)

Banker, Charnes, and Cooper (1984) extended the CCR model with a convexity constraint (Σλ = 1), allowing the frontier to exhibit increasing, constant, or decreasing returns to scale. BCC scores are always greater than or equal to CCR scores — it is a less restrictive model.

```swift
let result = try solver.solve(dmus: dmus, model: .bcc)
```

Use BCC when entities operate at different scales and you want to compare them only against peers of similar size.

## Super-Efficiency (Andersen-Petersen)

Standard DEA clamps all efficient entities at a score of 1.0, which makes it impossible to rank among them. Super-efficiency removes the evaluated entity from its own reference set, allowing scores above 1.0.

```swift
let result = try solver.solve(
    dmus: dmus,
    model: .superEfficiency(base: .ccr)
)

// Efficient DMUs now score > 1.0
// Inefficient DMU scores are identical to standard DEA
for score in result.scores where score.efficiency > 1.0 {
    print("\(score.name): super-efficiency = \(score.efficiency)")
}
```

> Note: BCC super-efficiency can produce infeasible LPs for extreme-vertex DMUs. These are flagged via ``DMUScore/superEfficiencyInfeasible`` rather than throwing.

## Slacks-Based Measure (Tone 2001)

The SBM model directly optimizes dimension-specific slacks instead of applying a uniform radial scaling factor. This means it can identify that "your price is fine but your energy cost is 30% too high and your warranty is 1 year too short" — something radial models cannot express.

SBM is non-oriented: it simultaneously considers input reductions and output expansions. The orientation parameter is ignored.

```swift
let result = try solver.solve(
    dmus: dmus,
    model: .sbm(returnsToScale: .constant)
)

for score in result.scores {
    print("\(score.name): SBM efficiency = \(score.efficiency)")
    if let inSlacks = score.inputSlacks {
        print("  Input improvements needed: \(inSlacks)")
    }
    if let outSlacks = score.outputSlacks {
        print("  Output shortfalls: \(outSlacks)")
    }
}
```

## Matrix-Form Convenience API

When working with tabular data, the matrix-form API skips DMU struct construction:

```swift
let result = try solver.solve(
    inputs: [[50, 20], [30, 30], [60, 10], [55, 25]],
    outputs: [[100, 80], [90, 70], [110, 60], [85, 65]],
    names: ["A", "B", "C", "D"],
    model: .ccr
)
```

Names are optional — if omitted, DMUs are labeled "DMU_1", "DMU_2", etc.

## Async Parallel Solving

For large datasets (100+ DMUs), the async solver distributes LP solves across available cores:

```swift
let solver = AsyncDEASolver(maxConcurrency: 4)
let result = try await solver.solve(dmus: dmus, model: .ccr)
```

Results are deterministic regardless of concurrency level. The solver uses bounded concurrency to prevent resource exhaustion.

## Interpreting Results

### Efficiency Scores

- **1.0** — The entity is on the efficient frontier
- **< 1.0** — The entity is inefficient; the score indicates the fraction of current inputs needed to become efficient (input-oriented) or the fraction of potential output being achieved (output-oriented)
- **> 1.0** — Only in super-efficiency mode; indicates how far beyond the frontier this entity extends

### Reference Sets

Each inefficient entity's score includes a reference set — the efficient entities that define its comparison point, along with lambda weights:

```swift
for score in result.scores where !score.isEfficient {
    print("\(score.name) benchmarks against:")
    for ref in score.referenceSet {
        print("  \(ref.name) (weight: \(ref.weight))")
    }
}
```

### Improvement Targets

Input and output targets show what the entity needs to achieve to reach the frontier:

```swift
for score in result.scores where !score.isEfficient {
    if let targets = score.targetInputs {
        print("\(score.name) target inputs: \(targets)")
    }
}
```

## Model Selection Guide

| Scenario | Recommended Model |
| --- | --- |
| Similar-scale entities, overall ranking | CCR |
| Mixed-scale entities, fair comparison | BCC |
| Rank efficient entities against each other | Super-efficiency (CCR or BCC base) |
| Need per-dimension improvement targets | SBM |
| Large dataset (100+ entities) | Any model via ``AsyncDEASolver`` |
