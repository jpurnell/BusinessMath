# Choosing the Right Inventory Model

A practical guide to selecting the inventory model that fits your data, product, and business context.

## Overview

Different inventory situations call for different models. This tutorial walks through the decision process and shows how ``InventoryAdvisor`` automates it. By the end, you will understand *why* each model applies in a given context — not just *which* one to pick.

## Step 1: Is This a Single-Period Problem?

The first question is whether your product has a single selling window. If the product is perishable, seasonal, or tied to a specific event (think: Valentine's Day flowers, concert merchandise, daily-baked bread), you face the **newsvendor problem**.

**Use ``NewsvendorModel`` when:**
- The product cannot be carried over to the next period
- You know the per-unit cost of having too many (overage) and too few (underage)
- Demand follows an approximately normal distribution

```swift
// Fresh produce: $5 profit per unit sold, $2 loss per unit unsold
let result = try NewsvendorModel<Double>.optimalQuantity(
    meanDemand: 200.0,
    demandStdDev: 40.0,
    underageCost: 5.0,
    overageCost: 2.0
)
// Critical fractile: 5/(5+2) = 0.714
// Order quantity: 200 + 0.566 × 40 ≈ 223 units
```

If the product is perishable but you don't have cost data, ``InventoryAdvisor`` will flag this and fall back to the reorder point model while recommending you gather cost information.

## Step 2: What Data Do You Have?

For continuous-replenishment products, the choice depends on what data you can provide:

### Minimal Data: Demand History + Lead Time

With just historical demand and a known lead time, use the **demand-only** safety stock method. This is the starting point for most small businesses.

```swift
let result = try ReorderPointModel<Double>.calculate(
    demandHistory: last30DaysOfSales,
    leadTime: 7.0,
    serviceLevel: 0.95
)
```

### Better: Add Lead Time Variability

If your supplier's delivery time varies (and it usually does), providing the lead time standard deviation switches to the **demand-and-lead-time** method, which accounts for both sources of uncertainty.

```swift
let result = try ReorderPointModel<Double>.calculate(
    demandHistory: last30DaysOfSales,
    leadTime: 7.0,
    serviceLevel: 0.95,
    leadTimeStdDev: 2.0,
    method: .demandAndLeadTime
)
```

This almost always produces a *higher* reorder point — and that is correct. Ignoring lead time variability systematically underestimates risk.

### Advanced: Forecast RMSE

If you run a demand forecast (for example, using Holt-Winters or exponential smoothing), the **forecast error** method uses the forecast's root mean square error instead of raw demand variability. This is more accurate when your forecast captures trend and seasonality that raw σ_d would overstate.

```swift
let ss = try SafetyStockModel<Double>.safetyStock(
    method: .forecastError,
    serviceLevel: 0.95,
    averageDemand: 100.0,
    demandStdDev: 15.0,
    leadTime: 7.0,
    forecastRMSE: 8.0
)
```

## Step 3: Should You Simulate?

The analytical formulas assume normally distributed demand. Real demand is often:
- **Skewed** (more small days than large days)
- **Bimodal** (weekday vs. weekend patterns)
- **Heavy-tailed** (occasional large orders)

With **30+ demand observations**, ``InventorySimulator`` builds an empirical distribution that captures these patterns without distributional assumptions:

```swift
let simResult = try InventorySimulator.simulate(
    demandHistory: last90DaysOfSales,
    meanLeadTime: 7.0,
    serviceLevel: 0.95,
    strategy: .empirical,
    iterations: 10_000,
    seed: 42
)
```

**When to use analytical vs. simulation:**

| Scenario | Recommendation |
|---|---|
| < 30 demand observations | Analytical only |
| 30-60 observations, normal-looking demand | Either; simulation validates analytical |
| 60+ observations, non-normal demand | Simulation preferred |
| Need reproducible results | Simulation with a fixed seed |

## Step 4: Do You Need EOQ?

The reorder point tells you *when* to order. The ``EOQModel`` tells you *how much*. If you have ordering costs (fixed cost per purchase order) and holding costs (storage, insurance, capital cost per unit per year), EOQ completes the picture:

```swift
let eoq = try EOQModel<Double>.calculate(
    annualDemand: 36_500.0,     // 100 units/day × 365
    orderingCost: 50.0,         // $50 per purchase order
    holdingCostPerUnit: 2.0     // $2/unit/year to store
)
print("Order \(eoq.orderQuantity) units every \(eoq.daysBetweenOrders) days")
```

## Let the Advisor Decide

``InventoryAdvisor`` encodes these decision rules. Give it what you know, and it tells you which model to use and why:

```swift
let profile = InventoryAdvisor.DataProfile(
    demandHistory: dailySales,
    leadTimeMean: 7.0,
    leadTimeStdDev: 2.0,
    annualDemand: 36_500.0,
    orderingCost: 50.0,
    holdingCostPerUnit: 2.0
)
let rec = InventoryAdvisor.recommended(for: profile)

// rec.recommendedModel         → .reorderPoint
// rec.safetyStockMethod        → .demandAndLeadTime
// rec.simulationRecommended    → true
// rec.eoqApplicable            → true
// rec.reasoning                → ["Lead time variability detected...", ...]
```

Each recommendation includes reasoning strings that explain the decision in plain language — useful for building user-facing explanations in your application.

## Common Pitfalls

1. **Ignoring lead time variability**: Most suppliers have variable lead times. Using the demand-only method when lead time varies underestimates the reorder point.

2. **Using too little data for simulation**: Empirical bootstrapping with < 30 observations can produce unreliable results because the sample poorly represents the true distribution.

3. **Confusing service level with fill rate**: A 95% cycle service level means you expect to *not* stock out in 95% of replenishment cycles. This is different from the fill rate (fraction of demand satisfied from stock).

4. **Applying newsvendor to replenishable goods**: The newsvendor model is for single-period decisions. For products you continuously restock, use the reorder point model.
